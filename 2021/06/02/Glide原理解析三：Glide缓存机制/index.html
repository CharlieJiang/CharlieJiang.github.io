    Glide原理解析三：Glide缓存机制

    发表于 2021-06-02 更新于 2021-06-03 分类于 Glide ， Android ， 第三方框架原理解析 阅读次数：

    缓存机制

    多级缓存

    Glide的缓存大致可以分为两个级别：内存缓存和磁盘缓存。如果要细分的话，又可以分为四个级别：活动资源缓存、内存缓存、资源缓存、原始数据缓存。其中前两者属于内存缓存，后两者属于磁盘缓存。

    • 活动资源缓存ActiveResources：弱引用缓存当前正在使用的资源对象
    • 内存缓存 LruResourcesCache：缓存最近使用过的资源对象
    • 资源缓存 Resource Cache：缓存经解码等加工操作原始数据后产生的资源
    • 原始数据缓存Data Cache：缓存从网络下载的，或是本地文件、Uri读取的原始数据

    缓存key

    缓存key是Glide中缓存资源的唯一性标识。以EngineKey为例：

    // EngineKey.java
class EngineKey implements Key {
  // 定义多个缓存Key的标识字段
  private final Object model;// 图片资源类型
  private final int width;// 图片宽度
  private final int height;// 图片高度
  private final Class<?> resourceClass;
  private final Class<?> transcodeClass;
  private final Key signature;// 签名
  private final Map<Class<?>, Transformation<?>> transformations;
  private final Options options;
  private int hashCode;
  ...

  @Override
  public boolean equals(Object o) {// 重写equal()方法，进行全字段比较
    if (o instanceof EngineKey) {
      EngineKey other = (EngineKey) o;
      return model.equals(other.model)
          && signature.equals(other.signature)
          && height == other.height
          && width == other.width
          && transformations.equals(other.transformations)
          && resourceClass.equals(other.resourceClass)
          && transcodeClass.equals(other.transcodeClass)
          && options.equals(other.options);
    }
    return false;
  }

  @Override
  public int hashCode() {// 重写hashCode()方法，定义自己的hashCode计算方式
    if (hashCode == 0) {
      hashCode = model.hashCode();
      hashCode = 31 * hashCode + signature.hashCode();
      hashCode = 31 * hashCode + width;
      hashCode = 31 * hashCode + height;
      hashCode = 31 * hashCode + transformations.hashCode();
      hashCode = 31 * hashCode + resourceClass.hashCode();
      hashCode = 31 * hashCode + transcodeClass.hashCode();
      hashCode = 31 * hashCode + options.hashCode();
    }
    return hashCode;
  }
...
}

    EngineKey中定义了图片资源类型model（如File、Uri、URL等）、图片宽高（显示图片的view的宽高）、签名等多个参数，来作为缓存Key的标识，并通过重写equal()和hashCode()方法保证key的唯一性。

    虽然缓存key设计的初衷是为了保证缓存资源的唯一性，避免重复缓存。但也因此会带来一些问题：

    1. 静态URL无法刷新问题：如果URL对应的图片资源发生变换而URL不变，则Glide不会下载网络图片，而是加载缓存中图片。这就会导致不会显示新的图片。
    2. 动态URL重复缓存问题：如果一张图片的URL是动态变化的，那么Glide每次加载图片时都会重新缓存这张图片。这就会导致资源的浪费。

    解决方案：

    • ①、上述两种问题都是资源本身的问题，服务端应确保不出现单一图片URL会发生变化、多张图片公用相同URL等问题，应该确保图片URL与图片资源匹配的唯一性。这也是最好的解决方式。
    • ②、关闭Glide的缓存功能。关闭Glide的缓存功能将使Glide的重要功能无法使用，将影响图片加载的体验、产生潜在的内存泄露等问题。
    • ③、修改签名signature：signature是Glide生成缓存key的参数之一，默认情况下signature是通过EmptySignature.obtain()生成的一个空Key，加载图片时，通过修改signature，就相当于变相修改了缓存key。但这样其实跟直接关闭Glide缓存功能类似，只是可能影响的范围会小点。

    活动资源缓存

    当一张图片资源正在被使用时，Glide首先会会将其放入ActiveResources中进行管理。ActiveResources部分源码如下：

    // ActiveResources.java
final class ActiveResources {
  private final boolean isActiveResourceRetentionAllowed;
  private final Executor monitorClearedResourcesExecutor;
  @VisibleForTesting final Map<Key, ResourceWeakReference> activeEngineResources = new HashMap<>();// 保存弱引用对象的键值对
  private final ReferenceQueue<EngineResource<?>> resourceReferenceQueue = new ReferenceQueue<>();// 弱引用对象的引用队列

  private ResourceListener listener;

  private volatile boolean isShutdown;
  @Nullable private volatile DequeuedResourceCallback cb;
  ...

    ActiveResources是一个支持引用计数的图片资源的弱引用集合。通过EngineKey对弱引用对象进行唯一性标识。ActiveResources中通过Map<Key,WeakReference<EngineResource<?>>>来存储管理图片资源的弱引用，并通过引用队列ReferenceQueue绑定弱引用对象，观察其gc事件，当弱引用对象被gc回收后，Glide也会从ActiveResourcces中将其移除。当弱引用对象的引用计数归零后，Glide会将其从ActiveResources中移除，同时将其放入内存缓存中。
    使用活动资源缓存能有效避免正在使用的图片资源被从内存缓存中回收，因为内存缓存是基于LRU算法的，而且一般内存缓存都会有一定的大小限制，当内存缓存满了以后，就可能会回收仍然在使用的图片。

    内存缓存

    如果图片资源最近被加载过，但当前没有被使用，则会将其放入内存缓存中。内存缓存部分源码如下：

    // LruResourceCache.java
public class LruResourceCache extends LruCache<Key, Resource<?>> implements MemoryCache

public class LruCache<T, Y> {
  private final Map<T, Entry<Y>> cache = new LinkedHashMap<>(100, 0.75f, true);// 双向链表存储图片缓存对象
  private final long initialMaxSize;
  private long maxSize;
  private long currentSize; 
  ...

    内存缓存是基于Glide自己实现的LruCache算法对图片资源进行管理，LruCache内部采用LinkedHashMap存储缓存对象的强引用，LinkedHashMap的双向链表结构，使得缓存的资源可以按加入顺序进行排序，可以应用LRU算法进行增删管理。通过EngineKey对内存缓存对象进行唯一性标识。当图片资源被从活动资源中移除时，会被放入到内存缓存中；当从内存缓存中加载图片资源时，会先将图片资源从内存缓存中移除，然后再放入活动资源中，防止图片资源因内存缓存已满而被从内存中回收。

    内存缓存主要是为了防止重复读取相同图片到内存中。

    磁盘缓存

    直接从网络、文件、Uri等获取的原始图片数据及原始图片数据经过初步解码（如裁剪、压缩等操作）的图片资源会被缓存到磁盘中。磁盘缓存的部分源码如下：

    // DiskLruCacheWrapper.java
public class DiskLruCacheWrapper implements DiskCache {
  private static final String TAG = "DiskLruCacheWrapper";

  private static final int APP_VERSION = 1;
  private static final int VALUE_COUNT = 1;
  private static DiskLruCacheWrapper wrapper;

  private final SafeKeyGenerator safeKeyGenerator;
  private final File directory;// 磁盘缓存目录
  private final long maxSize;// 磁盘缓存大小上限
  private final DiskCacheWriteLocker writeLocker = new DiskCacheWriteLocker();
  private DiskLruCache diskLruCache;// 磁盘缓存算法
  ...

    磁盘缓存同样采用DiskLruCache算法管理缓存资源。通过DataCacheKey对磁盘缓存对象进行唯一性标识。同时，磁盘缓存还分为多个缓存策略：

    • DiskCacheStrategy.RESOURCE：只缓存解码过的图片
    • DiskCacheStrategy.DATA：只缓存原始图片
    • DiskCacheStrategy.ALL： 既缓存原始图片，也缓存解码过的图片, 对于远程图片，缓存DATA和RESOURCE；对于本地图片，只缓存 RESOURCE。
    • DiskCacheStrategy.NONE：不使用磁盘缓存
    • DiskCacheStrategy.AUTOMATIC：默认策略，会对本地和远程图片使用最佳的策略；对下载网络图片，使用DATA；对于本地图片，使用RESOURCE

    磁盘缓存主要是防止重复从网络下载同一张图片。

    Bitmap复用池BitmapPool

    当不使用Glide的缓存功能时，从源地址加载图片并解析成Bitmap时，就需要用到BitmapPool。

    BitmapPool同样采用LRU算法管理复用的Bitmap。当一张图片要被解析成Bitmap时，会先从BitmapPool中查找可以被复用的Bitmap进行解析。
    当一个Bitmap因内存不足或复用池大小限制等原因被动地被从内存中移除时，其并不会被直接回收，而是会被加入BitmapPool中。只有从BitmapPool中被动地被移除的Bitmap，才会被真正地回收释放。
    BitmapPool并不会减少程序当前正在使用的内存大小，但BitmapPool可以减少频繁申请内存带来的内存抖动、碎片化等性能问题。

    Glide生命周期管理

    Glide在Glide.with(context)中就实现了生命周期管理，with根据传入的参数有不同的实现：

    // Glide.java
  //传入一个Context
  public static RequestManager with(@NonNull Context context)
  //传入一个activity
  public static RequestManager with(@NonNull Activity activity)
  //传入一个FragmentActivity
  public static RequestManager with(@NonNull FragmentActivity activity)
  //传入一个Fragment
  public static RequestManager with(@NonNull Fragment fragment)
  //传入一个View
  public static RequestManager with(@NonNull View view)

    上述生命周期管理可以分为两类：

    1. 传入一个ApplicationContext：Glide的生命周期就相当于绑定了整个应用，只要应用不退出，任何时候都能够加载，也可以理解为不对Glide生命周期进行管理。
    2. 传入activity、FragmentActivity 、Fragment 及View：这样就会创建一个看不见的fragment，Glide的生命周期就随着该Fragment的变化而变化。

    Glide与Activity的生命周期绑定

    • 创建RequestManagerFragment绑定到Activity上，RequestManagerFragment便可获知Activity的生命周期。
    • RequestManagerFragment内部持有ActivityFragmentLifecycle对象，用于转发RequestManagerFragment的生命周期事件：ActivityFragmentLifecycle内部持有一个生命周期监听者列表，在转发生命周期事件时，会遍历监听者列表依次进行转发。
    • 创建一个RequestManager，并将其注册到RequestManagerFragment.ActivityFragmentLifecycle上，以监听ActivityFragmentLifecycle的生命周期事件。因为RequestManager实现了LifecycleListener接口。这样就实现了RequestManager关联Activity的生命周期事件。

    问题

    1. 为什么使用了内存缓存，还要使用弱引用缓存？
      内存缓存使用LruCache算法实现，由于内存缓存大小会有限制，这种最近最少使用算法就可能会导致正在使用的图片被从内存缓存中回收。如果使用弱引用缓存正在使用的活动图片资源，就能避免此类问题。

    2. 弱引用缓存有大小限制吗？
      弱引用使用HashMap键值对缓存资源对象，缓存的大小取决于HashMap的容量大小。由于HashMap容量是会自增，所以弱引用缓存大小便没了限制。

    3. 磁盘缓存和内存缓存的Key一样吗？
      磁盘缓存是DataCacheKey:用于缓存原始数据。
      内存缓存是EngineKey：用于各种Loader（如HttpGlideUrlLoader、FileLoader等）从内存中加载资源。

    4. 内存缓存与磁盘缓存的缓存大小上限
      磁盘缓存：默认大小250M
      内存缓存：由MemorySizeCalculator根据屏幕尺寸与应用可用内存大小等综合进行计算所得。
