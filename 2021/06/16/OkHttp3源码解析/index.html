OkHttp3源码解析

发表于 2021-06-17 更新于 2021-08-01 分类于 Android ， OkHttp ， 第三方框架原理解析 阅读次数：

说明

本文基于OkHttp 3.4.1。

OkHttp请求的整体流程

从一次网络请求开始

一次简单的网络请求过程如下

// 1、创建OkHttpCLient对象
OkHttpClient okHttpClient = new OkHttpClient();
// 2、构造请求Request
Request request = new Request.Builder().url("https://www.google.com").build();
// 3、创建请求执行器Call
Call call = okHttpClient.newCall(request);
// 4、执行请求获取请求结果Response
Response response = call.execute();
// 5、解析响应
return response.body().string();

网络请求涉及的主要类

  • OkHttpClient：OkHttp的核心管理类，所有的内部逻辑和对象都归其管理。
  • Request：HTTP请求的封装类，包括url、method、header、body等常见请求参数。
  • Response：HTTP响应结果的封装类，包括code、message、header、body等常见响应参数。
  • Call：请求执行器，负责请求的调度、责任链处理等，具体的工作由RealCall来执行。

网络请求过程详细分析

1、创建OkHttpClient对象

OkHttpClient对象通过默认构造函数创建，构造函数内部通过构造器Builder执行初始化。

3、创建请求执行器RealCall

Call接口中定义了一些网络请求相关的操作，包括同步/异步执行请求、取消请求、请求的克隆等。 

public interface Call extends Cloneable {
  // 返回原请求
  Request request();

  // 同步执行请求
  Response execute() throws IOException;

  // 异步执行请求
  void enqueue(Callback responseCallback);
  
  // 取消请求
  void cancel();

  boolean isExecuted();// 请求是否已执行

  boolean isCanceled();// 请求是否已取消

  // 克隆请求
  Call clone();

  interface Factory {// Call的工厂
    Call newCall(Request request);
  }
}

Call对象通过OkHttpClient的newCall()方法来创建，newCall()方法内部通过RealCall.newRealCall()方法创建了一个RealCall对象（RealCall实现了Call接口）。所以实际上newCall()方法创建的就是RealCall对象，Call的实际工作也正是交由RealCall处理的。 

// OkHttpClient.java
public Call newCall(Request request) {
  return RealCall.newRealCall(this, request, false /* for web socket */);
}
// RealCall.java
static RealCall newRealCall(OkHttpClient client, Request originalRequest, boolean forWebSocket) {
  // 直接new一个RealCall返回
  RealCall call = new RealCall(client, originalRequest, forWebSocket);
  // 请求过程的事件监听，如DNS解析、socket链接、TLS连接等
  call.eventListener = client.eventListenerFactory().create(call);
  return call;
}

RealCall的构造函数中创建了一个请求拦截器RetryAndFollowUpInterceptor，用于处理请求的重试和重定向。

// RealCall.java
private RealCall(OkHttpClient client, Request originalRequest, boolean forWebSocket) {
  this.client = client;
  this.originalRequest = originalRequest;
  this.forWebSocket = forWebSocket;
  this.retryAndFollowUpInterceptor = new RetryAndFollowUpInterceptor(client, forWebSocket);// 重试和重定向拦截器
}

RealCall提供两种执行Request的方式，分别是同步方法execute()和异步方法enqueue()。因网络请求属于耗时操作，一般都是使用后者。RealCall.enqueue()源码： 

// RealCall.java
public void enqueue(Callback responseCallback) {
  synchronized (this) {// 代码块同步锁：保证请求只会执行一次
    if (executed) throw new IllegalStateException("Already Executed");
    executed = true;
  }
  captureCallStackTrace();
  eventListener.callStart(this);// 回调EventListener的callStart()方法
  // 创建异步线程AsyncCall去执行请求，并将异步线程放入分发队列中等待执行
  client.dispatcher().enqueue(new AsyncCall(responseCallback));
}

RealCall.enqueue()内部是通过创建一个异步线程AsyncCall对象来执行请求、处理请求回调，然后通过Dispatcher.enqueue()方法将AsyncCall对象交由Dispatcher执行。Dispatcher中持有多个任务队列，包括正在执行的同步任务队列、正在执行的异步任务队列及准备执行的异步任务队列。 

// Dispatcher.java
public final class Dispatcher {// 请求分发器
  private int maxRequests = 64;// 最大请求数
  private int maxRequestsPerHost = 5;
  private @Nullable Runnable idleCallback;

  /** Executes calls. Created lazily. */
  private @Nullable ExecutorService executorService;// 执行请求的线程池

  // 准备执行的异步任务队列
  private final Deque<AsyncCall> readyAsyncCalls = new ArrayDeque<>();

  // 正在执行的异步任务队列
  private final Deque<AsyncCall> runningAsyncCalls = new ArrayDeque<>();

  // 正在执行的同步任务队列
  private final Deque<RealCall> runningSyncCalls = new ArrayDeque<>();
  ...
}

Dispatcher.enqueue()方法根据具体情况会决定是立即执行新的异步任务，还是将其放入准备执行队列中。并最终通过Dispatcher中的线程池executorService去执行异步任务。 

// Dispatcher.java
synchronized void enqueue(AsyncCall call) {
  // 条件：运行中异步任务数量<最大请求数 & 同一Host下运行中异步任务数量<单Host最大请求数  
  if (runningAsyncCalls.size() < maxRequests && runningCallsForHost(call) < maxRequestsPerHost) {
    runningAsyncCalls.add(call);// 异步任务加入运行中异步任务队列
    executorService().execute(call);// 通过线程池执行异步任务
  } else {
    readyAsyncCalls.add(call);// 异步任务加入准备执行异步任务队列
  }
}

AsyncCall继承NamedRunnable，NamedRunnable实现了Runnable接口，并定义了execute()抽象方法。AysncCall通过实现NamedRunnable.execute()抽象方法来具体处理请求。 

// RealCall.java
protected void execute() {
  boolean signalledCallback = false;
  try {
    Response response = getResponseWithInterceptorChain();// 执行请求以获取响应结果
    if (retryAndFollowUpInterceptor.isCanceled()) {// 请求取消
      signalledCallback = true;
      responseCallback.onFailure(RealCall.this, new IOException("Canceled"));
    } else {// 请求成功
      signalledCallback = true;
      responseCallback.onResponse(RealCall.this, response);// 返回请求结果
    }
  } catch (IOException e) { // 请求异常处理
    if (signalledCallback) {
      // Do not signal the callback twice!
      Platform.get().log(INFO, "Callback failure for " + toLoggableString(), e);
    } else {
      eventListener.callFailed(RealCall.this, e);
      responseCallback.onFailure(RealCall.this, e);// 请求失败回调
    }
  } finally {
    client.dispatcher().finished(this);
  }
}

NamedRunnable.execute()中具体执行请求的方法是getResponseWithInterceptorChain()，RealCall的同步请求方法execute()也是通过getResponseWithInterceptorChain()方法获取请求结果。此方法也是整个请求的核心逻辑。 

// RealCall.java
Response getResponseWithInterceptorChain() throws IOException {
  // Build a full stack of interceptors.
  List<Interceptor> interceptors = new ArrayList<>();// 拦截器列表
  interceptors.addAll(client.interceptors());// 前置自定义拦截器
  interceptors.add(retryAndFollowUpInterceptor);// 重试和重定向拦截器
  interceptors.add(new BridgeInterceptor(client.cookieJar()));// 服务器交互拦截器
  interceptors.add(new CacheInterceptor(client.internalCache()));// 请求结果缓存拦截器
  interceptors.add(new ConnectInterceptor(client));// 连接建立拦截器
  if (!forWebSocket) {
    interceptors.addAll(client.networkInterceptors());// 后置自定义拦截器
  }
  interceptors.add(new CallServerInterceptor(forWebSocket));// 数据传输拦截器
  // 创建拦截器责任链串联所有拦截器
  Interceptor.Chain chain = new RealInterceptorChain(interceptors, null, null, null, 0,
      originalRequest, this, eventListener, client.connectTimeoutMillis(),
      client.readTimeoutMillis(), client.writeTimeoutMillis());
  // 通过拦截器责任链去执行请求(递归执行各拦截器完成请求)
  return chain.proceed(originalRequest);
}

getResponseWithInterceptorChain()方法中，将众多拦截器串联到拦截器责任链RealInterceptorChain上，并通过执行RealInterceptorChain.proceed()方法完成网络请求。 

// RealInterceptorChain.java
public Response proceed(Request request, StreamAllocation streamAllocation, HttpCodec httpCodec,
    RealConnection connection) throws IOException { 
  ...
  // 创建新的RealInterceptorChain，用于执行第index+1位置的拦截器（即拦截器列表中的下一个拦截器）

      connection, index + 1, request, call, eventListener, connectTimeout, readTimeout,

  Response response = interceptor.intercept(next);

  return response;
}

  RealInterceptorChain realChain = (RealInterceptorChain) chain;

RealInterceptorChain.proceed()方法与Interceptor.intercept()方法通过递归调用，依次执行拦截器列表中的各拦截器，完成了网络请求并获取请求结果。各拦截器的执行顺序与其加入拦截器列表的顺序一致：interceptors、RetryAndFollowUpInterceptor、BridgeInterceptor、CacheInterceptor、ConnectInterceptor、networkInterceptors、CallServerInterceptor。

总结：OkHttp将整个网络请求的复杂逻辑切分成一个个独立的模块，并命名为拦截器。然后通过责任链模式将拦截器串联在一起，通过依次执行拦截器来完成网络请求并获取响应结果。

OkHttp核心：拦截器

interceptors：自定义拦截器

interceptors被称为OkHttp中的Application Interceptor，因为它是拦截器责任链中的第一个拦截器，并且在一次网络请求中它只会被调用一次，其它拦截器则可能因为重试或重定向而被调用多次。

由于interceptors的特殊位置，它面对的是最原始的用户请求，它适合为请求添加一些公共参数，如应用版本、手机型号等，也可以为请求统一添加默认的请求头或自定义请求头。同时，interceptors也是拦截器责任链中响应结果最终回调到的位置，此处也适合做响应数据的处理，如json解析等。
为请求添加Application Interceptor的方式如下： 

OkHttpClient client = new OkHttpClient.Builder()
        .addInterceptor(new HttpLoggingInterceptor())
        .build();

RetryAndFollowUpInterceptor：重试和重定向拦截器

  • 重试：当请求内部抛出异常时，判断是否需要重试
    重试条件：
    client的retryOnConnectionFailure参数设置为false，不进行重试
    请求的body已经发出，不进行重试
    特殊的异常类型不进行重试（如ProtocolException，SSLHandshakeException等）
    没有更多的route（包含proxy和inetaddress），不进行重试
  • 重定向：当响应码是3xx重定向时，构建新的请求并发送

BridgeInterceptor：应用代码与服务器代码交互桥梁

BridgeInterceptor是应用程序层代码到网络层代码的桥梁。BridgeInterceptor负责基于用户请求构建服务端请求，基于服务端响应构建用户友好的响应。 

// BridgeInterceptor.java
public Response intercept(Chain chain) throws IOException {
  Request userRequest = chain.request();// 用户请求
  Request.Builder requestBuilder = userRequest.newBuilder();// 新请求构造器
  // ①、基于用户请求构建新的请求：完善用户请求的请求头
  RequestBody body = userRequest.body();
  if (body != null) {
    MediaType contentType = body.contentType();
    if (contentType != null) {
      requestBuilder.header("Content-Type", contentType.toString());
    }

    long contentLength = body.contentLength();
    if (contentLength != -1) {
      requestBuilder.header("Content-Length", Long.toString(contentLength));
      requestBuilder.removeHeader("Transfer-Encoding");
    } else {
      requestBuilder.header("Transfer-Encoding", "chunked");
      requestBuilder.removeHeader("Content-Length");
    }
  }

  if (userRequest.header("Host") == null) {
    requestBuilder.header("Host", hostHeader(userRequest.url(), false));
  }

  if (userRequest.header("Connection") == null) {
    requestBuilder.header("Connection", "Keep-Alive");
  }

  // gzip压缩支持
  boolean transparentGzip = false;
  if (userRequest.header("Accept-Encoding") == null && userRequest.header("Range") == null) {
    transparentGzip = true;
    requestBuilder.header("Accept-Encoding", "gzip");
  }

  List<Cookie> cookies = cookieJar.loadForRequest(userRequest.url());
  if (!cookies.isEmpty()) {
    requestBuilder.header("Cookie", cookieHeader(cookies));
  }

  if (userRequest.header("User-Agent") == null) {
    requestBuilder.header("User-Agent", Version.userAgent());
  }
  // ②、执行新请求，获取响应
  Response networkResponse = chain.proceed(requestBuilder.build());
  // 保存cookie
  HttpHeaders.receiveHeaders(cookieJar, userRequest.url(), networkResponse.headers());
  // ③、基于服务器响应构造新的用户友好的响应结果并返回
  Response.Builder responseBuilder = networkResponse.newBuilder()
      .request(userRequest);
  // 使用gzip解压原始响应数据
  if (transparentGzip
      && "gzip".equalsIgnoreCase(networkResponse.header("Content-Encoding"))
      && HttpHeaders.hasBody(networkResponse)) {
    GzipSource responseBody = new GzipSource(networkResponse.body().source());
    Headers strippedHeaders = networkResponse.headers().newBuilder()
        .removeAll("Content-Encoding")
        .removeAll("Content-Length")
        .build();
    responseBuilder.headers(strippedHeaders);
    String contentType = networkResponse.header("Content-Type");
    responseBuilder.body(new RealResponseBody(contentType, -1L, Okio.buffer(responseBody)));
  }

  return responseBuilder.build();
}

BridgeInterceptor完成了如下三项工作：

  • 1、基于用户请求构造新的完整请求，新的请求将用于发送至服务器请求数据
    检查用户请求，设置新请求的各Header属性：Content-Type、Content-Length、Transfer-Encoding、Host、Connection、Accept-Encoding、Cookie、User-Agent。其中Cookie的设置相对复杂些，需要借助CookieJar先从url中获取cookie列表（List）,然后按”name=value; name2=value2; “拼接各cookie，最后将拼接成的字符串设置为请求头cookie的值。 
    // BridgeInterceptor.java
List<Cookie> cookies = cookieJar.loadForRequest(userRequest.url());// 获取Cookie列表
if (!cookies.isEmpty()) {
  requestBuilder.header("Cookie", cookieHeader(cookies));// 设置cookie
}

private String cookieHeader(List<Cookie> cookies) {// 拼接Cookie列表
  StringBuilder cookieHeader = new StringBuilder();
  for (int i = 0, size = cookies.size(); i < size; i++) {
    if (i > 0) {
      cookieHeader.append("; ");
    }
    Cookie cookie = cookies.get(i);
    cookieHeader.append(cookie.name()).append('=').append(cookie.value());
  }
  return cookieHeader.toString();
}
  • 2、执行新请求，获取响应
    通过调用拦截器责任链的proceed()方法执行新构造的请求来获取服务器响应。
  • 3、基于服务器响应构造新的用户友好的响应结果并返回
    此处主要做的就是响应数据的GZIP解压处理

CacheInterceptor：请求缓存管理拦截器

CacheInterceptor负责请求缓存的读写处理，OkHttp中，缓存采用的DiskLruCache进行处理，缓存key就是经过处理的URL。

CacheInterceptor的处理逻辑如下：

// CacheInterceptor.java
public Response intercept(Chain chain) throws IOException {
  // 获取候选缓存  
  Response cacheCandidate = cache != null
      ? cache.get(chain.request())// 根据Request对象读取对应缓存
      : null;

  long now = System.currentTimeMillis();
   //  根据当前时间、请求对象、候选缓存创建缓存策略
  CacheStrategy strategy = new CacheStrategy.Factory(now, chain.request(), cacheCandidate).get();
  Request networkRequest = strategy.networkRequest;// 缓存策略中的网络请求，若为null则表示不使用网络
  Response cacheResponse = strategy.cacheResponse;//  缓存策略中的响应缓存，若为Null则表示不使用缓存

  if (cache != null) {
    cache.trackResponse(strategy);
  }
  //  候选缓存不为Null，而缓存响应为Null，表明不使用缓存，则需要关闭候选缓存，以防止内存泄露
  if (cacheCandidate != null && cacheResponse == null) {
    closeQuietly(cacheCandidate.body()); // The cache candidate wasn't applicable. Close it.
  }

  // 若不准使用网络，且缓存已过期，则直接返回504错误
  if (networkRequest == null && cacheResponse == null) {
    return new Response.Builder()
        .request(chain.request())
        .protocol(Protocol.HTTP_1_1)
        .code(504)
        .message("Unsatisfiable Request (only-if-cached)")
        .body(Util.EMPTY_RESPONSE)
        .sentRequestAtMillis(-1L)
        .receivedResponseAtMillis(System.currentTimeMillis())
        .build();
  }

  // 若不使用网络，则直接返回缓存响应
  if (networkRequest == null) {
    return cacheResponse.newBuilder()
        .cacheResponse(stripBody(cacheResponse))
        .build();
  }
  //  没有获取到缓存，则直接请求网络
  Response networkResponse = null;
  try {
    networkResponse = chain.proceed(networkRequest);
  } finally {
    // If we're crashing on I/O or otherwise, don't leak the cache body.
    if (networkResponse == null && cacheCandidate != null) {
      closeQuietly(cacheCandidate.body());
    }
  }

  // 如果网络响应和缓存响应都存在，则需要更新缓存策略中的缓存响应
  if (cacheResponse != null) {
    //  如果网络响应返回的code是304，表示网络响应没有body，需要更新缓存策略，并返回缓存响应  
    if (networkResponse.code() == HTTP_NOT_MODIFIED) {
      Response response = cacheResponse.newBuilder()
          // 合并缓存响应和网络响应的请求头，作为返回响应对象的请求头
          .headers(combine(cacheResponse.headers(), networkResponse.headers()))
          .sentRequestAtMillis(networkResponse.sentRequestAtMillis())// 发送请求时间
          .receivedResponseAtMillis(networkResponse.receivedResponseAtMillis())// 获取响应时间
          .cacheResponse(stripBody(cacheResponse))// 保存缓存响应
          .networkResponse(stripBody(networkResponse))// 保存网络响应
          .build();// 构建响应对象
      networkResponse.body().close();

      // Update the cache after combining headers but before stripping the
      // Content-Encoding header (as performed by initContentStream()).
      cache.trackConditionalCacheHit();
      cache.update(cacheResponse, response);// 更新缓存
      return response;
    } else {
      closeQuietly(cacheResponse.body());
    }
  }
  //  如果网络响应正常，则构建响应对象等待返回
  Response response = networkResponse.newBuilder()
      .cacheResponse(stripBody(cacheResponse))
      .networkResponse(stripBody(networkResponse))
      .build();

  if (cache != null) {
    // 缓存请求  
    if (HttpHeaders.hasBody(response) && CacheStrategy.isCacheable(response, networkRequest)) {
      // Offer this request to the cache.
      CacheRequest cacheRequest = cache.put(response);
      return cacheWritingResponse(cacheRequest, response);
    }
    // 如果请求的method不合法，则无法缓存请求，需要从缓存中移除请求
    if (HttpMethod.invalidatesCache(networkRequest.method())) {
      try {
        cache.remove(networkRequest);
      } catch (IOException ignored) {
        // The cache cannot be written.
      }
    }
  }

  return response;
}

private CacheStrategy getCandidate() {

  if (cacheResponse == null) {
    return new CacheStrategy(request, null);
  }

  }

  }
  // 请求的缓存控制不允许使用缓存

    return new CacheStrategy(request, null);

  CacheControl responseCaching = cacheResponse.cacheControl();
  if (responseCaching.immutable()) {
    return new CacheStrategy(null, cacheResponse);
  }

    }
    return new CacheStrategy(null, builder.build());

  String conditionName;
  String conditionValue;
  if (etag != null) {

    conditionValue = etag;
  } else if (lastModified != null) {
    conditionName = "If-Modified-Since";
    conditionValue = lastModifiedString;
  } else if (servedDate != null) {
    conditionName = "If-Modified-Since";
    conditionValue = servedDateString;
  } else {

  Internal.instance.addLenient(conditionalRequestHeaders, conditionName, conditionValue);
  // 构造一个请求，询问服务器资源是否过期
  Request conditionalRequest = request.newBuilder()
      .headers(conditionalRequestHeaders.build())
      .build();

### **ConnectInterceptor：DNS解析与Socket连接处理拦截器**  
ConnectInterceptor是OkHttp中重要的拦截器，它同时负责DNS解析和Socket连接（包括TLS连接）。ConnectInterceptor也简单，它只需要建立起到目标服务器的连接，剩下的操作交给下一个拦截器处理。

OkHttp与Volley区别

封装性：

  • Volley的封装性很好，使用非常方便
  • OkHttp的封装性较差，一般是再做一层封装后使用，如Retrofit框架就是基于OkHttp进行的封装

性能：

  • OkHttp是基于NIO（Okio）的，单请求性能上要比Volley快。
  • Volley在处理频繁的网络请求时性能要比OkHttp高：Volley为了提高请求的处理速度，采用了ByteArrayPool进行内存中的数据存储。（也正因此，Volley并不适合上传大文件）

扩展性：

  • Volley的扩展性很强，支持HttpClient、HttpUrlConnection、OkHttp

体积：

  • OkHttp偏大，大概有400+KB
  • Volley较小，大概100+KB

Retrofit

  • 基于OkHttp实现的Restful风格的封装。
  • 支持注解配置网络请求的方法、标记、参数
  • 支持多种HTTP Client实现网络请求，如OkHttp、HttpClient等
  • 支持同步、异步、RxJava
  • 可以配置不同的反序列化工具来解析数据，如json、xml等

思考

1、为什么BridgeInterceptor中要重新构造请求对象用于执行请求，而不是直接使用用户请求?

一个完整的HTTP请求报文需要包含请求行、请求头、请求体，我们通常使用OkHttp进行网络请求时设置的请求参数属于请求体，而BridgeInterceptor中重新构建Request的目的就是为用户请求添加上必要的请求头信息，以组成完整的报文进行网络请求。

2、为什么OkHttp中频繁使用建造者模式？

当一个类的构造函数中参数的个数超过4个，且这些参数中有些是可选，有些是必选时，传统有两种方式构建类的对象：

  • 一是通过多构造函数实现
  • 二是通过类似Java Bean的get/set方法实现

上述两种方式在构建类对象时容易产生问题：

  • 方式一多个构造函数会导致使用及阅读不便
  • 方式二在构建的过程中对象的状态容易发生变化，导致构建出错

建造者模式就是为了解决这些问题。构建者模式的实现方式：

  1. 在类A中创建一个静态内部类Builder，并将类A的所有参数复制到Builder中。
  2. 在类A中创建一个private的构造函数，入参为Builder
  3. 在Builder中创建一个以必选参数为入参的public构造函数
  4. 在Builder中创建可选参数的set方法，方法返回值为Builder本身，以实现链式调用
  5. 在Builder中创建一个build()函数，函数中创建一个类A的对象返回
    代码示例： 
    public class A {
    private int age;// 比选
    private String name;// 比选
    private String grade;// 可选
    private String clazz;// 可选

    private A(Builder builder){
        this.age = builder.age;
        this.name = builder.name;
        this.grade = builder.grade;
        this.clazz = builder.clazz;
    }

    public static class Builder{
        private int age;// 比选
        private String name;// 比选
        private String grade;// 可选
        private String clazz;// 可选

        public Builder(int age, String name){
            this.age = age;
            this.name = name;
        }

        public Builder setGrade(String grade) {
            this.grade = grade;
            return this;
        }

        public Builder setClazz(String clazz) {
            this.clazz = clazz;
            return this;
        }

        public A build(){
            return new A(this);
        }
    }

    public static void main(String[] args) {
        // 使用示例
        A a = new A.Builder(29,"cocoas")
                .setGrade("grade1")
                .setClazz("class1")
                .build();
    }
}

3、http的缓存机制

HTTP的缓存分为两种：强制缓存和对比缓存。强制缓存优于对比缓存。
强制缓存
客户端向服务器端请求数据时，服务端会返回数据及其过期时间。数据存入缓存数据库中，后续相同请求都会返回缓存的数据，直到缓存过期才会重新请求服务器。
对比缓存
客户端每次请求数据都需要与服务器交互，来确认缓存是否可用。首次请求服务器时，服务器会返回数据及缓存标识，客户端将其存入缓存数据库中。再次请求时，服务端会通过缓存标识校验缓存的有效性。当缓存依然有效时，服务端会返回304状态码，报文主体并不会返回；当缓存失效时，返回全部报文。

4、SSL协议与TLS协议的关系

SSL（Secure Socket Layer）安全套接字协议，是位于HTTP与TCP之间的协议加密层，基于HTTP标准并对TCP传递数据时进行加密处理。HTTP + SSL = HTTPS。
TLS（Transport Layer Security）传输安全层，是更为安全的升级版SSL。
