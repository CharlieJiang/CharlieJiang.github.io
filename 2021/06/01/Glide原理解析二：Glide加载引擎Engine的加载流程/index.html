Glide原理解析二：Glide加载引擎Engine的加载流程

发表于 2021-06-01 更新于 2021-06-02 分类于 Glide ， Android ， 第三方框架原理解析 阅读次数：

Engine的主要数据结构

分析Engine的加载流程前，先看下Engine的主要数据结构：

// Engine.java
// Jobs是EngineJob的管理类，负责对缓存EngineJob的管理
private final Jobs jobs;
// 负责创建缓存的key的工厂类
private final EngineKeyFactory keyFactory;
// DecodeJob工厂类。DecodeJob负责从原始资源或缓存中解码资源，并可执行变换和转码处理
private final DecodeJobFactory decodeJobFactory;
// EngineJob工厂类，EngineJob负责处理DecodeJob的解码结果（EngineJob实现了DecodeJob.Callback接口）
private final EngineJobFactory engineJobFactory;
// 回收资源：当资源被移除或释放时
private final ResourceRecycler resourceRecycler;
// 内存缓存：内存缓存中资源的增删管理，内部是LruCache缓存
private final MemoryCache cache;
// 磁盘缓存：内部是DiskLruCache缓存
private final LazyDiskCacheProvider diskCacheProvider;
// 活动资源缓存：内部通过弱引用实现缓存，优先级要高于内存缓存
private final ActiveResources activeResources;

Engine.load()方法加载资源分析

在分析SingleRequest时，SingleRequest在onSizeReady()方法中通过调用Engine.load()方法去加载处理资源。load()的源码如下：

// Engine.java
public <R> LoadStatus load(
    GlideContext glideContext,
    Object model,
    Key signature,
    int width,
    int height,
    Class<?> resourceClass,
    Class<R> transcodeClass,
    Priority priority,
    DiskCacheStrategy diskCacheStrategy,
    Map<Class<?>, Transformation<?>> transformations,
    boolean isTransformationRequired,
    boolean isScaleOnlyOrNoTransform,
    Options options,
    boolean isMemoryCacheable,
    boolean useUnlimitedSourceExecutorPool,
    boolean useAnimationPool,
    boolean onlyRetrieveFromCache,
    ResourceCallback cb,
    Executor callbackExecutor) {
  long startTime = VERBOSE_IS_LOGGABLE ? LogTime.getLogTime() : 0;
    // 创建资源内存缓存的key：由资源类型、签名、图片长宽等参数决定唯一性标识
  EngineKey key =
      keyFactory.buildKey(
          model,
          signature,
          width,
          height,
          transformations,
          resourceClass,
          transcodeClass,
          options);

  EngineResource<?> memoryResource;
  synchronized (this) {
      // 从内存中加载资源
    memoryResource = loadFromMemory(key, isMemoryCacheable, startTime);
    // 内存中未找到资源：等待从磁盘缓存EngineJob中读取，或是重新创建Job加载资源
    if (memoryResource == null) {
      return waitForExistingOrStartNewJob(
          glideContext,
          model,
          signature,
          width,
          height,
          resourceClass,
          transcodeClass,
          priority,
          diskCacheStrategy,
          transformations,
          isTransformationRequired,
          isScaleOnlyOrNoTransform,
          options,
          isMemoryCacheable,
          useUnlimitedSourceExecutorPool,
          useAnimationPool,
          onlyRetrieveFromCache,
          cb,
          callbackExecutor,
          key,
          startTime);
    }
  }

  // Avoid calling back while holding the engine lock, doing so makes it easier for callers to
  // deadlock.
  cb.onResourceReady(
      memoryResource, DataSource.MEMORY_CACHE, /* isLoadedFromAlternateCacheKey= */ false);
  return null;
}

load()方法做了两件事：构建资源的key；根据key加载资源。加载资源分三种方式：一是直接从内存中查找加载；二是从磁盘缓存中查找加载；三是需要重新加工原始资源。先看下loadFromMemory()方法： 

// Engine.java
private EngineResource<?> loadFromMemory(
    EngineKey key, boolean isMemoryCacheable, long startTime) {
  if (!isMemoryCacheable) {
    return null;
  }
    // 从活动资源中加载指定资源
  EngineResource<?> active = loadFromActiveResources(key);
  if (active != null) {
    if (VERBOSE_IS_LOGGABLE) {
      logWithTimeAndKey("Loaded resource from active resources", startTime, key);
    }
    return active;
  }
    // 从缓存中加载指定资源
  EngineResource<?> cached = loadFromCache(key);
  if (cached != null) {
    if (VERBOSE_IS_LOGGABLE) {
      logWithTimeAndKey("Loaded resource from cache", startTime, key);
    }
    return cached;
  }

  return null;
}

loadFromMemory()方法先尝试从活动资源ActiveResources中加载指定的资源，如果没有查找到，则会继续尝试从内存缓存中加载。继续看下loadFromActiveResources()和loadFromCache()的源码：

// Engine.java
private EngineResource<?> loadFromActiveResources(Key key) {
  EngineResource<?> active = activeResources.get(key);
  if (active != null) {
    active.acquire();
  }
  return active;
}

private EngineResource<?> loadFromCache(Key key) {
  EngineResource<?> cached = getEngineResourceFromCache(key);
  if (cached != null) {
    cached.acquire();
    activeResources.activate(key, cached);// 将从缓存中读取的资源加入到ActiveResources中
  }
  return cached;
}
private EngineResource<?> getEngineResourceFromCache(Key key) {
  Resource<?> cached = cache.remove(key);// 读取缓存中资源并从内存缓存中将其删除

  final EngineResource<?> result;
  ...
  return result;
}

loadFromActiveResources()方法通过ActiveResources获取活动资源缓存，loadFromCache()方法在获取到缓存资源后，先将缓存资源从内存缓存中删除，然后将资源加入到了ActiveResources中。这里ActiveResources的作用就是作为内存缓存的补充，因为内存缓存是有固定大小的，使用ActiveResources可以使内存缓存中存放更多之前使用过的图片。同时因ActiveResources是基于弱引用实现的，并不会造成内存上的过多占用。
再回看load()方法中内存缓存加载失败后调用的waitForExistingOrStartNewJob()方法：

// Engine.java
private <R> LoadStatus waitForExistingOrStartNewJob(
    省略参数...) {
    // 先从磁盘缓存中获取资源
  EngineJob<?> current = jobs.get(key, onlyRetrieveFromCache);
  if (current != null) {
    current.addCallback(cb, callbackExecutor);// 获取成功后需要进一步转码
    if (VERBOSE_IS_LOGGABLE) {
      logWithTimeAndKey("Added to existing load", startTime, key);
    }
    return new LoadStatus(cb, current);
  }
    // 磁盘缓存中未获取到资源：重新创建Job解码原始资源并处理
  EngineJob<R> engineJob = // 创建EngineJob：处理DecodeJob的解码结果
      engineJobFactory.build(
          key,
          isMemoryCacheable,
          useUnlimitedSourceExecutorPool,
          useAnimationPool,
          onlyRetrieveFromCache);

  DecodeJob<R> decodeJob = // 创建DecodeJob：解码原始资源
      decodeJobFactory.build(
          glideContext,
          model,
          key,
          signature,
          width,
          height,
          resourceClass,
          transcodeClass,
          priority,
          diskCacheStrategy,
          transformations,
          isTransformationRequired,
          isScaleOnlyOrNoTransform,
          onlyRetrieveFromCache,
          options,
          engineJob);

  jobs.put(key, engineJob);// EngineJob加入Jobs缓存管理

  engineJob.addCallback(cb, callbackExecutor);
  engineJob.start(decodeJob);// 开始执行DecodeJob

  if (VERBOSE_IS_LOGGABLE) {
    logWithTimeAndKey("Started new load", startTime, key);
  }
  return new LoadStatus(cb, engineJob);
}

waitForExistingOrStartNewJob()方法做了两件事：从磁盘缓存中读取资源；创建Job重新加载原始数据。读取磁盘缓存资源是通过缓存的EngineJob实现的，重新加载图片的原始数据也需要EngineJob去启动DecodeJob解码处理原始数据。看下engineJob.start(decodeJob)方法：

// EngineJob.java
public synchronized void start(DecodeJob<R> decodeJob) {
  this.decodeJob = decodeJob;
  // 根据是否从磁盘缓存加载资源选择不同的线程池去管理DecodeJob
  // 磁盘缓存加载资源：选择DiskCacheExecutor；
  // 原始数据加载资源：选择活动资源的GlideExecutor线程池
  GlideExecutor executor = 
      decodeJob.willDecodeFromCache() ? diskCacheExecutor : getActiveSourceExecutor();
  executor.execute(decodeJob);// 将DecodeJob放入线程池中准备执行
}

继续看下DecodeJob是如何执行的:

// DecodeJob.java
public void run() {
  ...
    runWrapped();
  ...
}

private void runWrapped() {
  switch (runReason) {
    case INITIALIZE:// 首次获取资源处理
        // 根据当前Stage及磁盘缓存的策略,获取下一阶段的Stage，Stage表示加载数据的数据源
      stage = getNextStage(Stage.INITIALIZE);
      currentGenerator = getNextGenerator();// 根据Stage创建对应的资源生成器
      runGenerators();
      break;
    case SWITCH_TO_SOURCE_SERVICE:// 磁盘缓存获取失败后，转而获取原始数据
      runGenerators();
      break;
    case DECODE_DATA:// 获取资源成功，进行资源加载
      decodeFromRetrievedData();
      break;
    default:
      throw new IllegalStateException("Unrecognized run reason: " + runReason);
  }
}

private Stage getNextStage(Stage current) {
  switch (current) {
    case INITIALIZE:
      return diskCacheStrategy.decodeCachedResource()// 数据源为缓存的解码资源
          ? Stage.RESOURCE_CACHE
          : getNextStage(Stage.RESOURCE_CACHE);
    case RESOURCE_CACHE:
      return diskCacheStrategy.decodeCachedData()// 数据源为缓存的原始数据
          ? Stage.DATA_CACHE
          : getNextStage(Stage.DATA_CACHE);
    case DATA_CACHE:
      // Skip loading from source if the user opted to only retrieve the resource from cache.
      return onlyRetrieveFromCache ? Stage.FINISHED : Stage.SOURCE;
    case SOURCE:
    case FINISHED:
      return Stage.FINISHED;
    default:
      throw new IllegalArgumentException("Unrecognized stage: " + current);
  }
}

继续看下获取资源生成器的方法getNextGenerator()：

// DecodeJob.java
private DataFetcherGenerator getNextGenerator() {
  switch (stage) { 
    case RESOURCE_CACHE:// 数据源为经过处理的图片的缓存数据
      return new ResourceCacheGenerator(decodeHelper, this);
    case DATA_CACHE:// 数据源为原始图片的缓存数据
      return new DataCacheGenerator(decodeHelper, this);
    case SOURCE:// 数据源为原始数据，如URL、File、Uri等
      return new SourceGenerator(decodeHelper, this);
    case FINISHED:// 资源已加载完成
      return null;
    default:
      throw new IllegalStateException("Unrecognized stage: " + stage);
  }
}

getNextGenerator()方法根据要加载数据源的类型RESOURCE_CACHE、DATA_CACHE、SOURCE，对应生成了各自的数据类型的处理器ResourceCacheGenerator、DataCacheGenerator、SourceGenerator。假设首次加载数据，则getNextGenerator()生成的就是SourceGenerator。接着下面的runGenerators()方法会调用SourceGenerator的startNext()方法：

// SourceGenerator.java
public boolean startNext() {
  if (dataToCache != null) {// dataToCache表示已经获取到的待缓存原始数据
    Object data = dataToCache;
    dataToCache = null;
    cacheData(data); // 缓存数据到磁盘  
  }
    // 上一步cacheData()会生成一个DataCacheGenerator，并赋值给sourceCacheGenerator
    // 这里就可以执行DataCacheGenerator.startNext()加载磁盘缓存数据。
  if (sourceCacheGenerator != null && sourceCacheGenerator.startNext()) {
    return true;// 到这里说明已经获取到原始数据并进行缓存处理，不再需要去获取原始数据
  }
  sourceCacheGenerator = null;

  loadData = null;
  boolean started = false;
  // 下面会根据缓存策略和各个ModelLoader去加载对应的原始数据
  while (!started && hasNextModelLoader()) {
    loadData = helper.getLoadData().get(loadDataListIndex++);
    if (loadData != null
        && (helper.getDiskCacheStrategy().isDataCacheable(loadData.fetcher.getDataSource())
            || helper.hasLoadPath(loadData.fetcher.getDataClass()))) {
      started = true;
      startNextLoad(loadData);
    }
  }
  return started;
}

上述方法中，相关类、接口说明：

  • ModelLoader：ModelLoader是一个工厂接口，用于将任意复杂的数据模型转换为具体数据类型，从而使DataFetcher能够获取由该模型表示的数据。
  • LoadData：由一系列资源的key和一个DataFetcher组成的数据结构
  • DataFetcher：检索数据以用于加载资源，如检索未缓存的原始数据并用于后续加载处理
  • HttpGlideUrlLoader：ModelLoader的实现类，其内部会构建一个HttpUrlFetcher对象，用于将GlideUrl（即URL）对应的资源转换为输入流。
  • FileLoader：ModelLoader的实现类，内部会创建一个FileFetcher对象，用于从文件中加载数据。

其中的HttpGlideUrlLoader、FileLoader是在Glide初始化时统一注册的：

HttpGlideUrlLoader通过内部的HttpUrlFetcher的loadData()方法加载网络数据：

// HttpUrlFetcher.java
public void loadData(
    @NonNull Priority priority, @NonNull DataCallback<? super InputStream> callback) {
  long startTime = LogTime.getLogTime();
  try {
    InputStream result = loadDataWithRedirects(glideUrl.toURL(), 0, null, glideUrl.getHeaders());
    callback.onDataReady(result);
  } catch (IOException e) {
    ...
    callback.onLoadFailed(e);
  } finally {
    ...
  }
}

private InputStream loadDataWithRedirects(
    URL url, int redirects, URL lastUrl, Map<String, String> headers) throws HttpException {
  ...
  // 构建并配置HttpUrlConnection
  urlConnection = buildAndConfigureConnection(url, headers);

  try {
    // Connect explicitly to avoid errors in decoders if connection fails.
    urlConnection.connect();
    // Set the stream so that it's closed in cleanup to avoid resource leaks. See #2352.
    stream = urlConnection.getInputStream();
  } catch (IOException e) {
    throw new HttpException(
        "Failed to connect or obtain data", getHttpStatusCodeOrInvalid(urlConnection), e);
  }

  if (isCancelled) {// 请求被取消
    return null;
  }
    // 请求结果处理
  final int statusCode = getHttpStatusCodeOrInvalid(urlConnection);
  if (isHttpOk(statusCode)) { // 请求成功
    return getStreamForSuccessfulRequest(urlConnection);
  } else if (isHttpRedirect(statusCode)) {// 请求重定向：构建重定向后的资源URL，再次通过loadDataWithRedirects()请求网络数据
    String redirectUrlString = urlConnection.getHeaderField(REDIRECT_HEADER_FIELD);
    if (TextUtils.isEmpty(redirectUrlString)) {
      throw new HttpException("Received empty or null redirect url", statusCode);
    }
    URL redirectUrl;
    try {
      redirectUrl = new URL(url, redirectUrlString);
    } catch (MalformedURLException e) {
      throw new HttpException("Bad redirect url: " + redirectUrlString, statusCode, e);
    }
    // Closing the stream specifically is required to avoid leaking ResponseBodys in addition
    // to disconnecting the url connection below. See #2352.
    cleanup();
    return loadDataWithRedirects(redirectUrl, redirects + 1, url, headers);// 递归调用自己的方法
  } else if (statusCode == INVALID_STATUS_CODE) {
    throw new HttpException(statusCode);
  } else {
    // 抛出异常
  }
}

可以看到，Glide底层是通过HttpUrlConnection加载网络数据。
加载完网络数据，我们再看下SourceGenerator中对加载结果的回调处理：

// SourceGenerator.java
private void startNextLoad(final LoadData<?> toStart) {
  loadData.fetcher.loadData(// 开始加载数据并处理数据加载结果回调
      helper.getPriority(),
      new DataCallback<Object>() {
        @Override
        public void onDataReady(@Nullable Object data) {
          if (isCurrentRequest(toStart)) {
            onDataReadyInternal(toStart, data);
          }
        }

        @Override
        public void onLoadFailed(@NonNull Exception e) {
          if (isCurrentRequest(toStart)) {
            onLoadFailedInternal(toStart, e);
          }
        }
      });
}

void onDataReadyInternal(LoadData<?> loadData, Object data) {
  DiskCacheStrategy diskCacheStrategy = helper.getDiskCacheStrategy();
  if (data != null && diskCacheStrategy.isDataCacheable(loadData.fetcher.getDataSource())) {
    dataToCache = data;// 将加载的网络数据缓存起来
    // 此处的cb就是DecodeJob(DecodeJob实现了DataFetcherGenerator.FetcherReadyCallback接口)
    // 即调用DecodeJob.reschedule()方法，DecodeJob会再次执行
    cb.reschedule();
  } else {
    cb.onDataFetcherReady(
        loadData.sourceKey,
        data,
        loadData.fetcher,
        loadData.fetcher.getDataSource(),
        originalKey);
  }
}

由于DecodeJob再次执行，会触发SourceGenerator.startNext()再次执行，此时SourceGenerator.dataToCache中已有数据，会继续执行SourceGenerator.cacheData()方法并生成DataCacheGenerator对象，然后会执行DataCacheGenerator.startNext()方法。DataCacheGenerator中通过FileLoader.FileFetcher加载被缓存在磁盘中的原始数据。DataCacheGenerator内部通过DataFetcher.DataCallback接收FileFetcher加载的数据，再通过DataCallback.onDataReady()中调用FetcherReadyCallback.onDataFetcherReady()将数据传递给FetcherReadyCallback，即传递给SourceGenerator（原因查看SourceGenerator.cacheData()）。SourceGenerator使用相同的方式将数据回传给DecodeJob，DecodeJob通过DecodeJob.Callback接口将数据回传给EngineJob，EngineJob通过ResourceCallback接口经过Engine将数据回传给SingleRequest。SingleRequest再通过事件回调将数据传递给与其绑定的ViewTarget，ViewTarget就可以为ImageView加载图片数据。

加载完网络数据后，整个数据的回传流程大概如下：

磁盘缓存——FileLoader.FileFetcher——DataCacheGenerator——SourceGenerator——DecodeJob——EngineJob——Engine——SingleRequest——ViewTarget——ImageView
