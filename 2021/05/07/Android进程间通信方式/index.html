Android进程间通信方式

发表于 2021-05-07 分类于 android ， 进程通信 阅读次数：

多种进程间通信方式

Android系统中存在多种方式用于实现进程间的通信，包括文件共享、Socket、Bundle、ContentProvider、BroadcastReceiver、Messenger、AIDL等。这些方式中，除了文件共享和Socket外，其余都是基于Binder实现的。

文件共享

  • 实现方式：两个进程通过读写同一个文件实现数据交换。
  • 使用场景：数据同步要求不高的进程间通信。
  • 局限：
    并发读写问题
    数据的完整性和时效性问题

Socket

Socket是一种基于网络套接字的进程间通信方式，通信双方通过建立起C/S模型进行数据交换。类似于Messenger方式。

Bundle

  • 实现方式：
    Bundle实现了Parceable接口，使得Bundle可以用于进程间数据传递。
    Android 的Activity、Service、Receiver均支持在Intent中传递Bundle数据，所以Bundle可以借助于Intent在各组件间跳转时传递数据。当各组件不在同一进程时，Bundle便实现了进程间的数据传递。
  • 使用场景：从当前进程中唤起其它进程的Activity、Service、Receiver时，可以通过Bundle传递需要发送给其它进程的数据。
  • 局限：
    传输数据限制：只支持基本数据类型或是序列化的数据类型。
    只能实现单向的简单数据传输。

ContentProvider

  • 实现方式：ContentProvider本身就是Android提供的不同应用间数据交换的API，ContentProvider通过指定的scheme+资源的路径，向外暴露数据的访问路径，其它应用可以基于资源访问路径构建ContentProvider，来实现远程进程资源的访问，并支持对资源的增删改查。Android系统提供的联系人、媒体文件等都是支持其它应用通过ContentProvider进行访问的。
  • 优点：
    一旦一个应用暴露出自己数据访问的scheme，不管此应用是否启动，其它应用均可通过该scheme访问暴露出来的数据。
    其它应用不仅能访问暴露出来的数据，还可以对数据进行增删改等操作。
  • 局限：
    应用需要自定义ContentProvider、自定义scheme，并在Manifest.xml中注册自定义的ContentProvider才能对外暴露自己的数据。

BroadcastReceiver

  • 实现方式：一个应用通过监听指定action的广播来读取另一个应用广播出来的数据。
  • 局限：广播是一种被动式的进程间通信方式，当一个应用向系统发送广播时，其它应用只能被动地接收广播数据。

Messenger

Messenger是基于消息Message的一种进程间通信方式，进程间通过交换各自的Messenger实现相互发送消息。Messenger是一种轻量级的IPC方案，它的底层实现是AIDL。

  • 实现方式：服务端+客户端

  • 服务端进程：
    首先创建一个service，在service中创建一个Handler处理消息，然后通过这个Handler构架一个Messenger对象，最后在service的onBind()中返回这个Messenger中的IBinder对象，客户端便可以利用返回的IBinder对象重新构建服务端进程的Messenger对象，并利用此Messenger对象访问服务端。
    若要向客户端进程发送消息，则可以在Handler处理消息时，从Message.replyTo中获取客户端进程的Messenger，并利用此Messenger向客户端进程发送消息。
    服务端实现代码： 

    // 服务端进程的服务
public class MessengerService extends Service {

    private final String TAG = "MessengerIPC";
    // Handler
    private Handler serverHandler;
    // Messenger
    private Messenger serverMessenger;
    // 服务端消息id
    private final int MSG_FROM_SERVER = 10088;
    // 服务端返回数据的key
    private final String SERVER_DATA_KEY = "serverDataKey";

    @Override
    public void onCreate() {
        super.onCreate();
        serverHandler = new Handler(){
            @Override
            public void handleMessage(@NonNull Message msg) {
                super.handleMessage(msg);
                // 处理客户端发送的消息
                Log.d(TAG,"收到来自客户端进程的消息");
                // 向客户端进程发送消息
                Message replyToClientMsg = Message.obtain(msg);
                // 构建要发送的数据
                Bundle data = new Bundle();
                data.putString(SERVER_DATA_KEY,"来自服务端进程的回复");
                replyToClientMsg.setData(data);
                replyToClientMsg.what = MSG_FROM_SERVER;
                try {
                    Thread.sleep(3000);
                    msg.replyTo.send(replyToClientMsg);
                } catch (RemoteException | InterruptedException e) {
                    e.printStackTrace();
                }
            }
        };
        // 通过Handler构建Messenger
        serverMessenger = new Messenger(serverHandler);

    }

    @Nullable
    @Override
    public IBinder onBind(Intent intent) {
        // 返回服务端Messenger的IBinder对象，客户端可根据此对象重新构建服务端的Messenger对象
        // 并通过此对象向服务端发送消息
        return serverMessenger.getBinder();
    }
}

// AndroidManifest.xml中注册服务
<service android:name=".MessengerService">
    <intent-filter>
            <action android:name="com.cocoas.messenger.service"></action>
        <category android:name="android.intent.category.DEFAULT"></category>
    </intent-filter>
</service>

  • 客户端进程：
    发送消息：首先通过ServiceConnection绑定服务端进程的service，根据service返回的IBinder对象创建服务端进程的Messenger对象，使用此Messenger对象向服务端进程发送消息。
    接收消息：创建Handler处理接收的消息，并基于此Handler构建客户端的Messenger对象，在向服务端进程发送消息时，将客户端进程的Messager对象放入Message.replyTo中随消息一起发送，服务端进程便可通过Message.replyTo引用到客户端进程的Messenger对象，并基于此向客户端进程发送消息。
    客户端实现代码： 

    public class MainActivity extends AppCompatActivity {

    private final String TAG = "MessengerIPC";
    // ServiceConnection
    private ServiceConnection serviceConn ;
    // 服务端 Messenger
    private Messenger serviceMessenger;
    // 客户端 Handler
    private Handler clientHandler;
    // 客户端Messenger
    private Messenger clientMessenger;

    // 服务端消息id
    private final int MSG_FROM_SERVER = 10088;
    // 服务端返回数据的key
    private final String SERVER_DATA_KEY = "serverDataKey";

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        // 处理服务端返回消息
        clientHandler = new Handler(){
            @Override
            public void handleMessage(@NonNull Message msg) {
                super.handleMessage(msg);
                switch (msg.what){
                    case MSG_FROM_SERVER:
                        // 读取服务端消息
                        Bundle data = msg.getData();
                        String info = data.getString(SERVER_DATA_KEY);
                        Log.d(TAG,"服务端返回数据：" + info);
                }
            }
        };
        // 构建客户端Messenger
        clientMessenger = new Messenger(clientHandler);
        // 远程服务连接
        serviceConn  = new ServiceConnection() {
            @Override
            public void onServiceConnected(ComponentName name, IBinder service) {
                Log.d(TAG,"onServiceConnected");
                // 构建服务端Messenger对象
                serviceMessenger = new Messenger(service);
                // 发送消息
                Message msg = Message.obtain(null,1);
                msg.replyTo = clientMessenger;
                try {
                    Thread.sleep(3000);
                    serviceMessenger.send(msg);
                } catch (RemoteException | InterruptedException e) {
                    e.printStackTrace();
                }
            }

            @Override
            public void onServiceDisconnected(ComponentName name) {

            }
        };
        bindService();
    }

    /**
     * 绑定远程服务
     */
    private void bindService(){
        Intent serviceIntent = new Intent();
        serviceIntent.setAction("com.cocoas.messenger.service");
        serviceIntent.setPackage("com.cocoas.ipcmessagerserver");
        bindService(serviceIntent,serviceConn,BIND_AUTO_CREATE);
    }

    /**
     * 解绑远程服务
     */
    private void unBindService(){
        unbindService(serviceConn);
    }

    @Override
    protected void onDestroy() {
        super.onDestroy();
        unBindService();
    }
}

  • 局限性：
    不适用高并发场景：Messenger内部通过Handler实现消息处理，消息处理是串行的方式，多条消息只能逐一处理。
    仅适合跨进程传递消息：Messenger主要就是用来传递消息，无法做到跨进程方法调用等需求。

AIDL

AIDL即Android Interface Definition Language，是一种Android接口定义语言，使用AIDL可以生成用于Android设备上两个进程间进行通信的代码。

  • 使用场景：多客户端、多线程并发访问。
    当允许多客户端访问应用提供的服务，并且需要处理多线程问题时，才需要使用AIDL。如果不需要处理多线程问题，可以选择使用封装好的Messenger。
  • 优点：客户端进程可以直接调用服务端进程暴露出来的方法。
  • 原理：
    AIDL IPC机制是面相接口的，AIDL使用代理类在客户端进程和服务端进程间传递数据。
    AIDL规定了两个进程之间如何使用Binder进行通讯，AIDL文件就是接口和数据声明。
    Android 使用我们写好的AIDL文件生成基于Binder的可以实现进程间通信的java代码，并在打包APK时将生成的Java代码打包进去，AIDL文件不会被打包。
  • 实现方式：
    服务端进程：
      创建AIDL接口描述文件，如IMyAidlInterface.aidl。
      实现AIDL接口：构建项目生成AIDL文件对应的java接口类IMyAidlInterface.java和内部抽象类IMyAidlInterface.Stub，通过实现Stub抽象类以实现AIDL接口。
      创建Service以向外暴露AIDL接口：创建Stub抽象类的实例来实现AIDL接口，并在onBind()中返回Stub实例对象以供客户端进程调用AIDL接口。
    客户端进程：
      创建AIDL接口描述文件IMyAidlInterface.aidl：与服务端AIDL文件相同。
      构建项目生成AIDL文件对应的接口类IMyAidlInterface.java和内部抽象类IMyAidlInterface.Stub。
      连接远程service：使用ServiceConnection连接远程服务，连接成功后，使用IMyAidlInterface.Stub.asInterface()方法将获取到的Ibinder对象转换为AIDL接口类IMyAidlInterface，然后就可以调用服务端AIDL方法。
  • 支持的数据类型：
    JAVA基本数据类型： int, long, double, char, boolean等。
    String和CharSequence
    List：AIDL实际接收的是ArrayList
    Map：AIDL实际接收的是HashMap
    序列化数据：实现了Parcelable或Serializable接口的数据
