原生多模块间的页面跳转实现方式：隐式Intent

隐式启动依赖于我们在AndroidManifest.xml中定义activity时通过intent-filter设置的匹配规则。在intent-filter中，我们可以设置三种匹配规则，分别是action、category、data。一个隐式Intent只有同时匹配了这三个规则，才能成功启动目标页面。

使用场景

应用之间跳转、H5与原生间交互、应用内部模块间跳转。

匹配规则

  • 单一intent-filter匹配：通常一个activity可以设置多个intent-filter，一般只需要匹配成功一个intent-filter，即可成功跳转目标页面。
  • 成套action、category、data匹配：若要匹配成功intent-filter，需要同时匹配成功其中的action、category、data。一个intent-filter中可以设置多个action、category、data，当intent-filter中存在多个action、category、data时，隐式Intent只需匹配成功intent-filter中的任意一组action、category、data即可。

  需要注意的是，使用隐式Intent启动activity时，如果没有设置category，startActivity()或startActivityForResult()方法会自动为我们添加一个默认category “android.intent.category.DEFAULT“。所以通常我们只要在intent-filter中添加这个默认的category，使用隐式Intent时就无需再设置category。即只需要匹配intent-filter中的action和data。

目标页面隐式跳转配置

  • 配置跳转的URL Scheme：基于intent-filter中的data匹配规则，配置隐式跳转所需的URL Scheme。
      什么是URL Scheme：Android中的scheme是一种页面内跳转协议，通过定义自己的scheme协议，可以非常方便跳转app中的各个页面；通过scheme协议，服务器可以定制化告诉App跳转那个页面，可以通过通知栏消息定制化跳转页面，可以通过H5页面跳转其他页面等。
      URL Scheme格式：scheme://host:post/path?params
        URL Scheme中包括协议Scheme、主机host、端口post、页面路径path及页面跳转参数params。
      如何设置URL Scheme： 

    <activity android:name="com.cocoas.module3.Module3Activity">
    <intent-filter>
        <data
            android:scheme="app"
            android:host="com.cocoas.module3"
            android:port="80"
            android:path="/activity"
            />
    </intent-filter>
</activity>

  • 配置默认的action和category：
    一般情况下，只要设置好默认的action和category，再配置好data，即配置好URL Scheme，即可实现隐式页面跳转。 

    <activity android:name="com.cocoas.module3.Module3Activity">
    <intent-filter>
        <!--必须的action：根据用户的数据类型打开响应的activity-->
        <action android:name="android.intent.action.VIEW"/>
        <!--必须：表示可通过浏览器中的链接跳转该页面-->
        <category android:name="android.intent.category.BROWSABLE"/>
        <!--必须：表示该页面可以被隐式启动-->
        <category android:name="android.intent.category.DEFAULT"/>
    </intent-filter>
</activity>

  • 实现隐式跳转:
      原生跳转

    Uri uri = Uri.parse("app://com.cocoas.module3:80/activity");
Intent intent = new Intent(Intent.ACTION_VIEW,uri);
startActivity(intent);

      H5链接跳转 

    <a href="app://com.cocoas.module3:80/activity">打开页面</a>

缺点

  • 每个页面都需要定义自己的intent-filter，比较繁琐。同时所有的页面规则都需要定义在Manifest文件中，导致扩展性和可维护性比较差，也不利于团队协作。
  • 跳转过程无法控制，如果通过intent-filter匹配的页面找不到，应用会直接崩溃。

路由为什么能够实现模块间的跳转

ARouter通过@Router注解将页面与页面路径之间的映射关系保存到一张路由表中，其它模块页面通过页面路径从路由表中查找到目标页面，然后就可以通过Intent实现跳转。

ARouter路由跳转原理

ARouter通过自定义注解设置页面访问路径、优先级、参数等信息；
编译时使用注解处理器APT生成用于存储页面路径path和activity.class映射关系的类文件，即路由表；
app进程启动时，加载这些类文件并将映射关系的数据读取到内存中；
进行路由跳转时，通过路由路径查找到目标页面的class，通过class构建跳转Intent，然后执行跳转。

ARouter常用自定义注解

  • @Route：声明路由路径path、分组group、优先级priority、额外信息extras等路由节点信息。
  • @Autowired：声明页面跳转携带的参数，用于页面间参数传递
  • @Interceptor：声明拦截器，用于拦截跳转并增加额外事件处理

使用注解处理器Processor解析注解

  • 多种Processor：Processor是javax中的注解处理器类，ARouter通过声明继承了Processor的RouteProcessor来处理@Route注解，类似的还有InterceptorProcessor、AutowiredProcessor用于处理Interceptor、Autowired注解。
  • @Route注解处理：遍历被@Route注解的节点，使用RouteMeta保存节点信息。根据分组名称保存节点信息到groupMap中。

JavaPoet生成Java类

JavaPoet是square公司开源的一款Java代码生成框架，JavaPoet根据注解可以很方便地生成Java源文件。
ARouter中，通过遍历groupMap，使用JavaPoet生成分组信息Java类和表信息Java类。

  • 路由节点信息类RouteMeta：RouteMeta是用于保存路由信息的Java类，通过解析@Route注解获取。RouteMeta类的结构如下：
    public class RouteMeta {
    //路由页面的类型：Activity或者IService
    public enum Type {
        ACTIVITY,
        ISERVICE
    }

    private Type type;
    //节点 (Activity)
    private Element element;
    //注解使用的类对象
    private Class<?> destination;
    //路由地址
    private String path;
    //路由分组
    private String group;
    ...
}
  • 分组信息类：根据路由路径的分组名（即@Route注解中的group参数）生成一个实现IRouteGroup接口的Java类，用来保存每个分组的路由信息。如：
    public class ARouter$$Group$$order implements IRouteGroup {
  @Override
  public void loadInto(Map<String, RouteMeta> atlas) {
    atlas.put("/order/RealOrderActivity", RouteMeta.build(RouteType.ACTIVITY, RealOrderActivity.class, "/order/realorderactivity", "order", null, -1, 1));
    atlas.put("/order/RefundOrderListActivity", RouteMeta.build(RouteType.ACTIVITY, RefundOrderListActivity.class, "/order/refundorderlistactivity", "order", null, -1, 1));
  }
}
  • 表信息类：根据模块名称生成一个实现了IRouteRoot接口的Java类，用来保存所有的分组信息类。如：
    public class ARouter$$Root$$order_list implements IRouteRoot {
  @Override
  public void loadInto(Map<String, Class<? extends IRouteGroup>> routes) {
    routes.put("order", ARouter$$Group$$order.class);
    routes.put("real", ARouter$$Group$$real.class);
  }
}

路由表Warehouse

路由表用于保存应用所有模块中的路由信息，包括分组信息类，路由页面、IService对象、拦截器等。

如何生成路由表：遍历所有的dex，查找所有使用JavaPoet生成的分组信息类和表信息类，并将分组信息类加载到Warehouse.groupsIndex中，在实际跳转时加载对应分组的路由信息。

Postcard

ARouter调用build()方法查询路由表返回的对象就是PostCard，Postcard继承自RouteMeta，用于保存路由节点的路径、分组名、传递的数据等信息，并可发出实际的跳转操作navigation。

  • 传递数据：Postcard.withXXX()
  • 页面跳转：Postcard.navigation()
  • 页面跳转动画：PostCard.withTransition()

拦截器Interceptor

Android中使用startActivity()执行页面跳转后，页面跳转的过程则不再受控。ARouter实现了在页面跳转过程中添加一些自定义的功能，比如添加参数、判断是否需要登录等，是一种AOP思想的实现。

  • 拦截器声明：拦截器使用@Interceptor注解声明，并可设定拦截器的优先级priority和名称name。
  • 实现拦截器：拦截器实现类需要实现拦截器接口IInterceptor，并实现其process()方法和init()方法。
  • 生成拦截器映射类：项目编译时，JavaPoet会根据模块名称生成一个实现了IInterceptorGroup接口的Java类，用于保存模块中所有的拦截器实现类。如：
    public class ARouter$$Interceptors$$commonutil implements IInterceptorGroup {
  @Override
  public void loadInto(Map<Integer, Class<? extends IInterceptor>> interceptors) {
    interceptors.put(2, RouterInterceptor.class);
  }
}
  • 加载拦截器映射类到Warehouse中：APP进程启动时ARouter会将拦截器映射关系分组加载到Warehouse.interceptorsIndex中，并在需要时进行拦截器实例化。

ARouter优点

  • 基于注解实现了路由映射关系自动注册与分布式路由管理。
  • 编译期间基于注解处理器实现注解解析，并生成路由映射文件，而非使用反射，不会影响运行时性能。
  • 映射关系按组分类、多级管理，按需初始化。
  • 灵活的降级策略，每次跳转都会返回跳转结果，避免原生跳转失败导致的应用崩溃。
  • 自定义拦截器对路由跳转进行拦截，实现跳转过程中的可控干预，如参数插入、登录判断等处理。
  • 支持依赖注入，可实现跨模块API调用。
  • 支持直接解析标准URL进行跳转，并自动注入参数到目标页面中。
